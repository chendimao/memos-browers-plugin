# Safari 临时安装测试打包实施计划

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 为现有浏览器扩展新增 Safari 临时安装测试打包能力，执行构建后同时产出 `dist/chrome`、`dist/firefox` 和 `dist/safari`。

**Architecture:** 继续沿用“共享基础 manifest + 浏览器差异补丁 + 统一构建脚本”的结构，在现有 Chrome/Firefox 双目标基础上扩展 Safari 目标。Safari 产物以可在 macOS Safari 中临时安装测试为目标，构建阶段增加 Safari manifest 补丁与产物校验，运行时为可能缺失的扩展 API 提供能力探测与保护。

**Tech Stack:** Vite 5、Vue 3、WebExtension manifest、Node.js 构建脚本、Safari Web Extensions。

---

## 功能目标

- 新增 `dist/safari` 构建产物目录。
- 新增 `npm run build:safari` 命令。
- `npm run build` 默认同时生成 Chrome、Firefox、Safari 三个生产包目录。
- Safari 产物用于 macOS Safari 的临时安装测试。
- 尽量保持与 Chrome/Firefox 一致的扩展结构与功能入口。

## 技术方案 / 架构说明

- 基于现有 `manifest/base.json`、`manifest/chrome.json`、`manifest/firefox.json` 扩展出 `manifest/safari.json`。
- 复用 `scripts/build-extension.mjs` 生成三目标产物，静态资源与 Vite 页面构建继续共用。
- 升级 `scripts/verify-build-output.mjs`，将 Safari 产物与 Safari manifest 约束纳入校验。
- 对 `background.js` 与 `src/utils/extensionApi.js` 中的关键扩展 API 做特性探测，减少 Safari 不支持接口时的直接报错风险。
- 补充一份 Safari 临时安装测试说明文档，说明如何使用 `dist/safari` 在 Safari 中做本地验证。

## 分步骤任务列表

### 任务 1：先补 Safari 构建回归校验

**Files:**
- Modify: `scripts/verify-build-output.mjs`
- Test: `dist/safari/manifest.json`

**Step 1: 写失败校验**

- 在校验脚本中把目标集合从 `chrome/firefox` 扩展到 `chrome/firefox/safari`。
- 增加 Safari 产物目录存在断言，以及 Safari manifest 关键字段断言。
- 先假定 Safari 需要至少包含：`manifest_version`、`name`、`version`、`permissions`、`host_permissions`、`action`、`icons`、`background`。

**Step 2: 运行校验确认失败**

Run: `npm run build:chrome && npm run verify:build`
Expected: FAIL，提示缺少 `dist/safari` 或 Safari manifest 相关字段。

**Step 3: 保持失败结果作为后续实现基线**

- 记录失败原因为“尚未实现 Safari 构建目标”，不在这一步提前修改生产代码。

**Step 2: 扩展构建脚本**

**Step 3: 新增 Safari manifest 补丁**

- 创建 `manifest/safari.json`。
- 仅加入 Safari 临时安装测试必须的补丁字段，避免复制整份完整 manifest。

**Step 4: 运行构建确认生成 Safari 目录**

Run: `npm run build:safari`
Expected: PASS，并生成 `dist/safari`。

## 潜在风险

- Safari 对部分 WebExtension API 的支持细节可能与 Chrome/Firefox 不完全一致，尤其是 `contextMenus`、`scripting.executeScript`、`action.openPopup`。
- 在没有 Xcode 的前提下，这次只能做到 Safari 临时安装测试路径，不能验证正式打包/签名链路。
- 如果 Safari 的 manifest 对某些字段有更严格要求，可能需要后续根据实际 Safari 报错继续补充。
- 当前项目自动化测试较少，本次主要依赖构建校验与手工验证说明降低回归风险。
