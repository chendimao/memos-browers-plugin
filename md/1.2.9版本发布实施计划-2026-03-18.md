# 1.2.9 版本发布实施计划-2026-03-18

## 功能目标

将当前项目版本从 `1.2.8` 升级到 `1.2.9`，补充明确的更新记录，完成构建验证，并发布到 GitHub Releases。

## 技术方案 / 架构说明

本次发布将基于当前 `master` 创建独立 worktree，在隔离分支中完成版本号更新与发布说明整理，避免影响主工作区中尚未提交的 README 改动。版本号会统一更新到构建与扩展元数据所需位置，并通过已有的 GitHub Actions Release 工作流在推送 tag 后自动生成发布产物。

更新记录不写入 README，避免与当前主工作区文档改动发生冲突；改为新增独立的中文发布记录文档，并在 GitHub Release 中使用明确的发布说明文本。

## 分步骤任务列表

### 任务 1：确认版本涉及文件与发布差异

**涉及文件：**
- 检查：`package.json`
- 检查：`manifest/base.json`
- 检查：`git log` / `git diff`（自 `1.2.8` 以来）

**步骤：**
1. 确认当前版本号位置。
2. 梳理自 `1.2.8` 以来的主要功能变更，提炼发布说明。
3. 确认将采用的发布 tag 形式与 Release 标题。

### 任务 2：更新版本号与发布记录

**涉及文件：**
- 修改：`package.json`
- 修改：`manifest/base.json`
- 新增：`md/1.2.9更新记录-2026-03-18.md`

**步骤：**
1. 将版本号统一更新为 `1.2.9`。
2. 新增独立更新记录文档，写明本次发布的用户可见改动。
3. 确保不修改主工作区中已有未提交的 README 文件。

### 任务 3：本地构建与发布前验证

**涉及文件：**
- 验证：`dist/**`
- 验证：`release-assets/**`

**步骤：**
1. 运行 `npm run build`。
2. 运行 `npm run verify:build`。
3. 运行 `npm run build:release-assets`。
4. 确认 `release-assets/` 中生成四个 `1.2.9` 的 zip 包。

### 任务 4：提交、打 tag 并发布到 GitHub

**涉及文件：**
- Git 提交与 tag
- GitHub Release

**步骤：**
1. 提交版本升级改动，提交信息使用中文。
2. 合并回 `master`。
3. 推送 `master` 到远端。
4. 创建并推送 `1.2.9` tag。
5. 等待 GitHub Actions Release 工作流完成。
6. 用 `gh release edit` 写入明确的更新记录并确认资产上传完成。

## 涉及文件

- `package.json`
- `manifest/base.json`
- `md/1.2.9版本发布实施计划-2026-03-18.md`
- `md/1.2.9更新记录-2026-03-18.md`

## 潜在风险

- 当前主工作区存在未提交的 `README.md` 与 `README.en.md` 改动，必须避免直接在原工作区发布。
- 如果 tag 与 `package.json` 版本不一致，Release workflow 会失败。
- 远端现有历史 tag 使用不带 `v` 的形式，本次也应保持一致，避免产生双版本标签混乱。
- 如果发布说明只依赖自动生成，可能无法准确体现本次用户可见更新，因此需要额外写明更新记录。
